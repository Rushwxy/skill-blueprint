# Skill Blueprint v3 — 能自我进化的元 Skill

## 设计哲学

> **先搜再建，生而能进。**
> 创建一个 Skill 前先搜有没有现成的；一个 Skill 出生后，能通过 9 维评分 + 单维改进 + 双评委审查的进化循环持续变好。

本 Skill 的核心任务：
1. **生成**：把模糊的自然语言翻译成标准 SKILL.md（预检层 → 骨架推导 → 确认写入）
2. **进化**：对已有 Skill 进行达尔文式定向优化（9 维评分 → 单维改进 → 审查通过/回滚）
3. **统一**：确保所有产出的 Skill 格式一致、agent 间可互读
4. **高效**：生成模式 1 轮确认，维护模式每轮只改 1 个维度

---

## 预检层：先搜再建（进入核心工作流前必执行）

> 仅「生成模式」走预检层。维护模式直接进入进化循环区。

### Layer 0: 典型度判断（1 秒决策）

| 典型（值得搜） | 不典型（跳过搜，直接新建） |
|------------|---------------------|
| 信息检索、监控、对比、日报周报 | 高度个人化的业务流程 |
| 数据分析、格式转换、代码生成 | 内网/私有系统专用场景 |
| Agent/Skill/MCP 等通用热门话题 | 用户明确说了"按我这个独特的逻辑" |
| 跨平台可复用的通用任务 | 涉及个人密码/密钥的非标准化操作 |

**决策规则**：典型 → 执行 Layer 1 搜索；不典型 → 直接跳到 Step 0；拿不准 → 倾向于搜。

### Layer 1: 搜索匹配（3 级级联）

| 级别 | 来源 | 搜索方式 |
|:--:|------|---------|
| 1 | **本地已安装 Skill** | 列出 `~/.workbuddy/skills/` 下的 Skill，匹配名称和描述 |
| 2 | **GitHub 开源 Skill** | WebSearch：`github.com <需求关键词> skill SKILL.md` |
| 3 | **通用 Web 搜索** | WebSearch：`<需求场景> skill workflow template` |

### Layer 1 结果处理

| 匹配度 | 判断标准 | 处理方式 |
|:------:|---------|---------|
| **完美匹配** | 功能覆盖 ≥ 90% | 告知用户 → 安装使用 → **结束** |
| **可改编** | 功能覆盖 50-90%，改编 < 新建 30% 成本 | 展示给用户选择：直接用还是新建 |
| **无匹配** | 没找到或改造成本 > 新建 30% | 进入核心工作流 Step 0 |

## 核心工作流（生成模式）

### Step 0: 意图解译 + 路由（0-1 轮）

**路由分支（优先判断）**：
- 用户说"帮我做/生成/创建一个 XX Skill" → **生成模式**（走下面 Step 1-3）
- 用户说"优化/改进/修复/升级 XX Skill" → **维护模式**（跳到进化循环区 evo step 1）
- 模糊不清时追问一句："你是要新建一个 Skill，还是优化现有的？"

**生成模式 — 意图提取**：从用户描述中提取核心要素。信息够就跳到 Step 1；缺则提最多 1 个确认问题。

| 维度 | 提取内容 | 示例 |
|------|---------|------|
| 任务本质 | 这个 Skill 解决什么问题 | "帮用户对比两个手机哪个该买" |
| 触发条件 | 什么时候该加载 | "用户说'哪个好'、'选哪个'" |
| 工作步数 | 从意图推导需要几步 | "搜信息 → 对比 → 推荐" |
| 约束条件 | 格式/来源/交互/偏好约束 | "必须引用视频评测" |

**规则**：信息 ≥ 3 项 → 直接进 Step 1；缺 1 项 → 问 1 个问题；缺多项 → 用 1 个概括性问题收窄。禁止一次抛 3+ 个问题。

---

### Step 1: 骨架自动推导

基于意图一次性完整输出骨架。**推导规则**：
- Step 数量 3-5 个，用户自己说了步骤数就听用户的
- 步骤间是递进关系，上一个 step 的输出是下一个的输入
- 分叉逻辑归到同一个 step 内处理

**骨架标准格式**：
```markdown
---
name: <skill-name>
description: |
  一句话说明 + 触发场景 + 触发词。使用第三人称。
agent_created: true
---

# <skill-name>

## 一句话概述

## 触发条件
- 触发词：[具体到场景，如"买XX哪个好"，而非孤立的"推荐"]

## 执行流程
### Step 1: <阶段名>
- 目标 / 输入 / 执行 / 输出 / 验收 / ⚠️ 别犯

## 全局约束
## 局限说明
```

---

### Step 2: Checkpoint 确认

展示完整骨架 + 问："步骤和方向对吗？要改什么直接说。"

| 用户回应 | 处理 |
|----------|------|
| "好/行/没问题" | → Step 3 写入 |
| "XX改成YY" | → 改后重新展示 |
| "不确定" | → 提供 2-3 个具体建议供选 |

---

### Step 3: 写入文件

写入到 `~/.workbuddy/skills/<skill-name>/SKILL.md`。
- YAML frontmatter 必须有 `agent_created: true`
- description 用第三人称
- 写入后告知路径 + 一句话总结核心能力

## 进化循环区（维护模式）

> 当 Step 0 路由到"维护模式"时触发。基于达尔文 2.0 的 9 维评分 + 单维改进 + 双评委审查循环。
> **核心原则**：每次只改 1 个维度的 1 个方面，改完验证，通过才保留。杜绝"全改一遍然后不知道哪个改动起了作用"。

### evo step 1: 基线评估

- **目标**：对目标 Skill 进行 9 维基线评分，建立改进前的参照基准。
- **执行**：
  1. 用 Read 工具读取目标 Skill 文件
  2. 按「附录：9 维评分标准」逐项打分（每维 0-10，总分 90）
  3. 记录评分到 `<skill-dir>/scores_history.jsonl`（每行一个 JSON：`{"timestamp":"...","version":"vX","scores":{...},"total":X}`）
  4. 如果 Skill 目录下有 `.git`，用 Bash 执行 `git add -A && git commit -m "evo-baseline: [name] score [total]"`
- **输出**：9 维评分表 + 总分 + 标注最低分维度
- **⚠️ 别犯**：凭印象打分不逐项对照标准 → 分数不可复现；不写评分文件直接进 evo step 2 → 回滚时丢失基线

---

### evo step 2: 单维度定向改进

- **目标**：从 9 维中选最低分维度，生成改进方案供用户确认。
- **执行**：
  1. 取最低分维度（并列时优先级：安全性 > 错误处理 > 完整性 > 其他）
  2. 生成改进方案：当前 X 分 → 目标 Y 分、具体改动位置和内容
  3. 用 AskUserQuestion 展示方案，用户确认后执行
  4. 执行前用 Bash 做 git commit 保存快照：`git commit -m "evo-improve: [name] [维度] X→Y"`
- **输出**：改进方案 + 用户确认
- **⚠️ 别犯**：一次改多个维度 → 无法归因效果；用户说"全改"就全改 → 进化循环的单维度原则被破坏

---

### evo step 3: 重新评分 + 保留/回滚

- **目标**：改进后重新打分，对比决定保留或回滚。
- **执行**：
  1. 对改进后的 Skill 重新 9 维评分
  2. **新总分 > 旧总分** → 接受：
     - Bash：`git commit -m "evo-accept: [name] +[差值]分 ([维度])"`
     - 追加评分记录到 scores_history.jsonl
  3. **新总分 ≤ 旧总分** → 回滚：
     - Bash：`git revert HEAD --no-edit`（禁止 git reset --hard）
     - 追加评分记录，标记 `"outcome": "reverted"`
- **⚠️ 别犯**：git reset --hard 当回滚 → 历史被改写；回滚后不记录评分 → 历史空白

---

### evo step 4: 双评委审查

- **目标**：2 个独立评委对改进后的 Skill 全维度评分，一致认可才通过。
- **执行**：
  1. 启动 2 个全新 LLM 角色评委（不得复用上一轮的评委身份，避免锚定效应）
  2. 每个评委独立对 9 维度打分（每维 0-10），给出总分 + 各维明细 + 综合判断理由
  3. **裁决规则**：
     - 2 个评委总分均 > 旧总分 → ✅ 通过，本轮进化结束
     - 至少 1 个评委总分 ≤ 旧总分 → 退回 evo step 2，换一个维度重试
     - 连续 2 轮不通过 → 放弃本轮，保留当前版本
     - 单轮总分涨幅 < 1 分 → 早停，保留当前版本
- **输出**：双评委判决表
  ```
  | 评委 | 可读性 | 完整性 | ... | 总分 | 旧总分 | 判决 |
  |------|:------:|:------:|-----|:----:|:----:|:----:|
  | 评委A | 7 | 8 | ... | 68 | 66 | ✅ |
  | 评委B | 8 | 8 | ... | 67 | 66 | ✅ |
  
  评委A 综合理由：...
  评委B 综合理由：...
  ```
- **⚠️ 别犯**：复用上一轮评委 → 锚定效应；连续 2 轮不通过还继续 → 无限循环

## 不做什么（反模式）

### 通用反模式
1. ❌ **不搜就建**——最大反模式。必须先过预检层
2. ❌ **搜到不好用还硬推**——匹配度 < 50% 就说没找到
3. ❌ **把用户当需求分析师**——不问"目标用户是谁""核心指标是什么"
4. ❌ **多轮拉锯**——预检层最多 1 轮 + 核心工作流最多 1 轮确认
5. ❌ **生成空壳**——每个 step 的"执行"必须有实质性内容
6. ❌ **越俎代庖**——用户说 4 步就 4 步
7. ❌ **过度设计**——搭骨架不写实现

### 进化循环专项反模式
8. ❌ **禁止自评**——生成者不得同时当评委；评分必须独立执行
9. ❌ **禁止 git reset --hard**——回滚只允许 git revert
10. ❌ **禁止进化循环跳跃**——evo 1→2→3→4 必须顺序执行
11. ❌ **禁止多维度同时改动**——evo step 2 只改 1 个维度
12. ❌ **禁止增量灌水**——不得通过降低评分标准来刷分

## 附录：典型骨架模板

### 信息检索型
```
Step 1: 拆解需求 → Step 2: 多源搜索 → Step 3: 结构化整理 → Step 4: 输出交付
```

### 分析决策型
```
Step 1: 理清问题 → Step 2: 收集数据 → Step 3: 对比分析 → Step 4: 给出结论
```

### 内容生成型
```
Step 1: 明确规格 → Step 2: 素材准备 → Step 3: 生成输出 → Step 4: 质检验收
```

### 自动化操作型
```
Step 1: 确认目标 → Step 2: 预检环境 → Step 3: 执行操作 → Step 4: 验证结果
```

### 评估评审型
```
Step 1: 确定标准 → Step 2: 逐项评估 → Step 3: 汇总结论 → Step 4: 改进建议
```

## 附录：9 维评分标准

> 用于进化循环的基线评估和双评委审查。每维 0-10 分，总分 90。

| # | 维度 | 0-3 分 | 4-6 分 | 7-8 分 | 9-10 分 |
|---|------|--------|--------|--------|---------|
| 1 | **可读性** | 结构混乱，术语不统一 | 有基本结构但部分模糊 | 结构清晰，术语统一 | 新人 10 分钟可完全理解 |
| 2 | **完整性** | 缺 YAML 或执行区 | 三区齐全但字段不足 | 三区齐全 + 自检清单 | 完全符合蓝图骨架全部要素 |
| 3 | **模块匹配度** | 多个 step 无工具调用 | 大部分有但 1-2 存疑 | 全部 step 工具匹配合理 | 工具选择精准，无冗余 |
| 4 | **错误处理** | 无任何异常处理描述 | 提到但无具体策略 | 关键 step 有异常分支 | 每个高风险 step 有降级策略 |
| 5 | **安全性** | 无安全约束 | 有基础约束未全覆盖 | 硬约束覆盖核心风险 | 三级风险定级完整 |
| 6 | **IO 闭合** | step 间输入输出断裂 | 基本闭合但 1-2 处隐式依赖 | 全部显式声明，链路可追踪 | 数据格式在各 step 间一致 |
| 7 | **反例覆盖** | 无任何反例 | 1-2 个但未覆盖高风险 step | 每个高风险 step 有反例 | 与全局反模式交叉验证无遗漏 |
| 8 | **约束一致性** | 约束间互相矛盾 | 无矛盾但缺优先级链 | 分类清晰 + 优先级链明确 | 含冲突消解规则 |
| 9 | **执行效率** | 冗余 step 2+ 个 | 有 1 个冗余 | step 数量合理，确认点适中 | 每步有实质产出，无流程 step |

## 已知局限

1. 生成的 Skill 是"骨架级"——具体工具调用细节需在实际使用中补全
2. 对含多分支决策树（10+ 路径）的复杂 Skill，建议先简化核心逻辑
3. 预检层搜索依赖关键词匹配，语义级近似可能漏掉
4. 进化循环的 9 维评分依赖 LLM 判断，非客观量化——双评委设计部分缓解但无法根除
5. 进化循环依赖 Git 做版本控制；如果 Skill 目录不在 Git 仓库中，回滚功能不可用
6. 单次进化只改 1 个维度——多维度优化需要多轮进化循环
