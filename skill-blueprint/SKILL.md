# Skill Blueprint v3 — 能自我进化的元 Skill

> **先搜再建，生而能进。**
> 以可承受的低效率维护所有人和 agent 协作时的认知一致性——半截工作交接不需要重构。

## 设计哲学

1. **先搜再建**：创建前先搜有没有现成的。多花一轮搜索，可能省掉 70-100% 新建成本。
2. **生而能进**：Skill 出生后能通过进化循环持续变好——9 维评分、单维改进、双评委审查。
3. **标准化优先**：所有产出遵循统一格式。宁可模板多写几行，也不要各写各的导致交接时重构。
4. **一次确认**：生成模式最多 1 轮确认。维护模式每轮只改 1 个维度，改完验证。

---

## 共识对齐

### 硬约束（不可违反）

- **蓝图骨架强制**：输出的 Skill 必须严格遵循标准骨架（YAML frontmatter → 一句话概述 → 触发条件 → 执行区 → 全局约束 → 局限说明），结构不可省略
- **预检必须**：进入核心工作流前必须先执行预检层，不得跳过
- **人审必须**：生成过程中至少包含 1 个人工确认点（骨架确认）；维护模式每轮改进需用户确认
- **输出路径安全**：生成的 Skill 文件禁止写入系统目录（C:\Windows、/System 等）或敏感配置目录（.ssh、.env）
- **禁止越权**：本 Skill 只负责生成/优化 Skill 文本，不负责执行生成的 Skill

### 软约束（默认遵守，用户可覆盖）

- 反例推荐填写但不强制（用户坚持跳过可跳过）
- 简单 Skill（≤3 步、信息充分）骨架可精简，但触发条件和验收标准不可省略
- 输出文件默认保存到 `~/.workbuddy/skills/<skill-name>/`

### 术语定义

- **预检层**：进入核心工作流前的搜索判断层——先搜有没有现成方案，再决定新建还是改编
- **骨架**：Skill 的标准结构模板（YAML frontmatter + 执行区 + 结束区）
- **进化循环**：对已有 Skill 进行 9 维评分 → 单维改进 → 重新评分 → 双评委审查的闭环
- **单维改进**：每次只改 9 维中 1 个维度的 1 个方面，改完验证，杜绝"全改一遍不知道哪个起效"

## 预检层：先搜再建（进入核心工作流前必执行）

> 仅「生成模式」走预检层。维护模式直接进入进化循环区。

### Layer 0: 典型度判断（1 秒决策）

| 典型（值得搜） | 不典型（跳过搜，直接新建） |
|------------|---------------------|
| 信息检索、监控、对比、日报周报 | 高度个人化的业务流程 |
| 数据分析、格式转换、代码生成 | 内网/私有系统专用场景 |
| Agent/Skill/MCP 等通用热门话题 | 用户明确说了"按我这个独特的逻辑" |
| 跨平台可复用的通用任务 | 涉及个人密码/密钥的非标准化操作 |

**决策规则**：典型 → 执行 Layer 1 搜索；不典型 → 直接跳到 Step 0；拿不准 → 倾向于搜。

### Layer 1: 搜索匹配（3 级级联）

| 级别 | 来源 | 搜索方式 |
|:--:|------|---------|
| 1 | **本地已安装 Skill** | 列出 `~/.workbuddy/skills/` 下的 Skill，匹配名称和描述 |
| 2 | **GitHub 开源 Skill** | WebSearch：`github.com <需求关键词> skill SKILL.md` |
| 3 | **通用 Web 搜索** | WebSearch：`<需求场景> skill workflow template` |

### Layer 1 结果处理

| 匹配度 | 判断标准 | 处理方式 |
|:------:|---------|---------|
| **完美匹配** | 功能覆盖 ≥ 90% | 告知用户 → 安装使用 → **结束** |
| **可改编** | 功能覆盖 50-90%，改编 < 新建 30% 成本 | 展示给用户选择：直接用还是新建 |
| **无匹配** | 没找到或改造成本 > 新建 30% | 进入核心工作流 Step 0 |

## 核心工作流（生成模式）

### Step 0: 意图解译 + 路由（0-1 轮）

**路由分支（优先判断）**：
- 用户说"帮我做/生成/创建一个 XX Skill" → **生成模式**（走下面 Step 1-3）
- 用户说"优化/改进/修复/升级 XX Skill" → **维护模式**（跳到进化循环区 evo step 1）
- 模糊不清时追问一句："你是要新建一个 Skill，还是优化现有的？"

**生成模式 — 意图提取**：从用户描述中提取核心要素。信息够就跳到 Step 1；缺则提最多 1 个确认问题。

| 维度 | 提取内容 | 示例 |
|------|---------|------|
| 任务本质 | 这个 Skill 解决什么问题 | "帮用户对比两个手机哪个该买" |
| 触发条件 | 什么时候该加载 | "用户说'哪个好'、'选哪个'" |
| 工作步数 | 从意图推导需要几步 | "搜信息 → 对比 → 推荐" |
| 约束条件 | 格式/来源/交互/偏好约束 | "必须引用视频评测" |

**规则**：信息 ≥ 3 项 → 直接进 Step 1；缺 1 项 → 问 1 个问题；缺多项 → 用 1 个概括性问题收窄。禁止一次抛 3+ 个问题。

---

### Step 1: 骨架自动推导

基于意图一次性完整输出骨架。**推导规则**：
- Step 数量 3-5 个，用户自己说了步骤数就听用户的
- 步骤间是递进关系，上一个 step 的输出是下一个的输入
- 分叉逻辑归到同一个 step 内处理

**骨架标准格式**：
```markdown
---
name: <skill-name>
description: |
  一句话说明 + 触发场景 + 触发词。使用第三人称（"当用户...时，应使用本 Skill..."）。
agent_created: true
---

# <skill-name>

## 一句话概述
[从用户意图推导，1-2 句说清做什么]

## 触发条件
- 触发词：[具体到场景，如"买XX哪个好"，而非孤立的"推荐"]
- 触发场景：[什么时候该加载]

## 执行流程

### Step 1: <阶段名>
- **目标**：[这一步要达成什么]
- **输入**：[需要什么信息/状态]
- **执行**：[具体怎么做，清晰到另一个 agent 不用猜]
- **输出**：[这一步产出什么]
- **验收**：[怎么判断做完了、做对了]
- **异常处理**：[常见异常 + 降级路径]
- **⚠️ 别犯**：[这一步最常见的错误/反例]

### Step 2: <阶段名>
...

## 全局约束
- [跨 step 的通用规则，如格式要求、语言风格、数据来源规范]

## 局限说明
- [当前已知的能力边界]
```

**关键质量标准**（生成时逐条自查）：
- ✅ 另一个 agent 读一遍就能执行——不需要猜"这一步到底要我做什么"
- ✅ "验收"不含糊——能量化的量化，不能量化的说清"什么样算完"
- ✅ 没有空洞词——不说"充分了解""全面分析"，说"收集至少 X 个信源"
- ✅ 触发词结合场景——不是孤立的"推荐"，而是"买XX哪个好"

---

### Step 2: Checkpoint 确认

展示完整骨架 + 问："步骤和方向对吗？要改什么直接说。"

| 用户回应 | 处理 |
|----------|------|
| "好/行/没问题" | → Step 3 写入 |
| "XX改成YY" | → 改后重新展示 |
| "不确定" | → 提供 2-3 个具体建议供选 |

---

### Step 3: 写入文件 + METADATA

写入到 `~/.workbuddy/skills/<skill-name>/SKILL.md`。
- YAML frontmatter 必须有 `agent_created: true`
- description 用第三人称
- 写入后告知路径 + 一句话总结核心能力

**每个生成的 Skill 文件末尾自动追加 METADATA 注释块**：
```markdown
<!-- METADATA
  GENERATED_BY: skill-blueprint v3.1
  GENERATED_AT: YYYY-MM-DD HH:MM
  SKILL_NAME: <name>
  EVOLUTION_CYCLE: 0
  SCORE_HISTORY: <skill-dir>/scores_history.jsonl
-->
```

## 进化循环区（维护模式）

> 当 Step 0 路由到"维护模式"时触发。基于达尔文 2.0 的 9 维评分 + 单维改进 + 双评委审查循环。
> **核心原则**：每次只改 1 个维度的 1 个方面，改完验证，通过才保留。

### evo step 1: 基线评估

- **目标**：对目标 Skill 进行 9 维基线评分，建立改进前的参照基准。
- **执行**：
  1. 用 Read 工具读取目标 Skill 文件
  2. 按「附录：9 维评分标准」逐项打分（每维 0-10，总分 90）
  3. 记录评分到 `<skill-dir>/scores_history.jsonl`
  4. 如果 Skill 目录下有 `.git`，Bash：`git add -A && git commit -m "evo-baseline: [name] score [total]"`
- **输出**：9 维评分表 + 总分 + 标注最低分维度
- **⚠️ 别犯**：凭印象打分 → 分数不可复现；不写评分文件 → 回滚丢失基线

---

### evo step 2: 单维度定向改进

- **目标**：从 9 维中选最低分维度，生成改进方案供用户确认。
- **执行**：
  1. 取最低分维度（并列时优先级：安全性 > 错误处理 > 完整性 > 其他）
  2. 生成改进方案：当前 X 分 → 目标 Y 分、具体改动位置和内容
  3. AskUserQuestion 展示方案，用户确认后执行
  4. Bash 保存快照：`git commit -m "evo-improve: [name] [维度] X→Y"`
- **输出**：改进方案 + 用户确认
- **⚠️ 别犯**：一次改多个维度 → 无法归因；用户说"全改"就全改 → 破坏单维原则

---

### evo step 3: 重新评分 + 保留/回滚

- **目标**：改进后重新打分，对比决定保留或回滚。
- **执行**：
  1. 对改进后的 Skill 重新 9 维评分
  2. **新总分 > 旧总分** → 接受：
     - Bash：`git commit -m "evo-accept: [name] +[差值]分 ([维度])"`
     - 追加评分记录到 scores_history.jsonl
  3. **新总分 ≤ 旧总分** → 回滚：
     - Bash：`git revert HEAD --no-edit`（禁止 git reset --hard）
     - 追加评分记录，标记 `"outcome": "reverted"`
- **⚠️ 别犯**：git reset --hard → 历史被改写；回滚不记录 → 历史空白

---

### evo step 4: 双评委审查

- **目标**：2 个独立评委全维度评分，一致认可才通过。
- **执行**：
  1. 启动 2 个全新 LLM 角色评委（不得复用上一轮，避免锚定效应）
  2. 每个评委独立对 9 维度打分（每维 0-10），给出总分 + 各维明细 + 综合理由
  3. **裁决规则**：
     - 2 个评委总分均 > 旧总分 → ✅ 通过，本轮进化结束
     - 至少 1 个评委总分 ≤ 旧总分 → 退回 evo step 2，换维度重试
     - 连续 2 轮不通过 → 放弃本轮，保留当前版本
     - 单轮总分涨幅 < 1 分 → 早停，保留当前版本
- **输出**：双评委判决表
- **⚠️ 别犯**：复用上一轮评委 → 锚定效应；连续 2 轮不通过还继续 → 无限循环

## 不做什么（反模式黑名单）

> Agent 更擅长推导"怎么做"，但需要明确"什么不该做"的边界。

### 通用开发反模式（8 条）
1. ❌ **不搜就建**——用户说需求后第一反应不能是"我来拆步骤"，必须先过预检层
2. ❌ **搜到不好用还硬推**——匹配度 < 50% 就说没找到，不硬塞
3. ❌ **把用户当需求分析师**——不问"目标用户是谁""核心指标是什么"，用户需要 Skill 不是 PRD
4. ❌ **多轮拉锯**——预检层最多 1 轮搜索 + 核心工作流最多 1 轮确认
5. ❌ **生成空壳**——每个 step 的"执行"部分有实质内容，不是"根据情况选择工具"这种空话
6. ❌ **越俎代庖**——用户说按 4 步来就 4 步，不强行改成 3 步
7. ❌ **过度设计**——本 Skill 搭骨架，不替用户决定每个 step 调哪个 API
8. ❌ **忽略异常处理**——每个高风险 step 必须有异常分支和降级路径，不能假设所有操作成功

### 进化循环专项反模式（5 条）
9. ❌ **禁止自评**——生成者不得同时当评委；评分必须由独立评委执行
10. ❌ **禁止 git reset --hard**——回滚只允许 git revert；reset --hard 会改写历史
11. ❌ **禁止进化循环跳跃**——evo 1→2→3→4 必须顺序执行；无基线评估就直接改进无法判断效果
12. ❌ **禁止多维度同时改动**——evo step 2 只改 1 个维度；同时改多个无法归因
13. ❌ **禁止增量灌水**——不得通过降低评分标准、放宽检查来刷分；分数提升必须来自实际质量改进

## 附录：5 种标准骨架模板

> 以下模板用于 Step 1 骨架推导时快速匹配用户意图类型。每种模板含完整 4 步结构和 1 条关键设计要点。

### 信息检索型
```
Step 1: 拆解需求 — 明确检索主题、范围、深度、时间窗
Step 2: 多源搜索 — 3-5 组不同关键词，覆盖不同信源，去重
Step 3: 结构化整理 — 筛选、分类、标注来源，按相关性/时效性排序
Step 4: 输出交付 — 生成结构化报告/列表，含来源链接和摘要
```
**关键设计要点**：step 2 至少 3 组不同关键词，确保信息广度；step 3 区分"多源验证"和"单一来源"。

### 分析决策型
```
Step 1: 理清问题 — 明确对比对象、决策维度、权重分配
Step 2: 收集数据 — 从多源拉取数据，统一格式和口径
Step 3: 对比分析 — 按维度逐项对比，量化打分，产出对比矩阵
Step 4: 给出结论 — 加权总分 + 优劣势总结 + 风险提示 + 可操作的建议
```
**关键设计要点**：step 1 必须明确定量标准，不能是"好不好"这种模糊词；step 3 每项对比附带依据。

### 内容生成型
```
Step 1: 明确规格 — 输出格式、篇幅、风格、数据来源、受众
Step 2: 素材准备 — 拉取和预处理原始数据/素材
Step 3: 生成输出 — 按规格和模板生成内容，分段产出
Step 4: 质检验收 — 格式检查、事实核查、风格一致性检查，输出最终版本
```
**关键设计要点**：step 1 规格必须具体可量化（如"不超过 1000 字""必须引用视频评测"）；step 4 不是"检查一下"而是明确检查项。

### 自动化操作型
```
Step 1: 确认目标 — 操作对象、操作类型、预期结果、约束条件
Step 2: 预检环境 — 检查前置条件（权限/依赖/路径），确认无误
Step 3: 执行操作 — 批量执行，带错误处理和日志记录
Step 4: 验证结果 — 展示操作统计，确认是否达到预期，归档日志
```
**关键设计要点**：step 2 的预检必须覆盖权限和依赖；step 3 的错误处理不能假设所有操作成功。

### 评估评审型
```
Step 1: 确定标准 — 评估维度、评分标准（每档对应什么表现）、权重
Step 2: 逐项评估 — 按标准对每个维度打分，附依据和证据
Step 3: 汇总结论 — 加权总分、优劣势、关键问题归类
Step 4: 改进建议 — 具体、可操作的改进项，优先级排序
```
**关键设计要点**：step 1 的标准必须能量化（如"可读性 7-8 分 = 结构清晰术语统一"）；step 4 的建议必须有优先级。

## 附录：异常处理约定

| 异常场景 | 处理策略 | 降级路径 |
|---------|---------|---------|
| 预检层搜索无结果 | 如实告知用户，直接进入 Step 0 从零构建 | 跳过预检层，不阻塞主流程 |
| Step 0 需求模糊无法解析 | 追问 1 轮澄清关键参数；仍不明则给出 2 个最可能解读供选择 | 避免多轮拉锯，最多 2 轮追问 |
| Step final 文件写入失败（权限/磁盘满） | 重试 1 次（换输出路径），仍失败则将 Skill 文本直接展示给用户 | 不静默丢弃产出物 |
| LLM 输出幻觉（工具名不存在、骨架不合法） | Step 1 生成后做工具名与可用工具交叉校验；不匹配时回退重填 | 自检→修正→重填，上限 2 次 |
| 用户中途取消或修改需求 | 已完成的中间产物不丢弃，作为下次对话的延续起点 | 保存到工作记忆 |
| 维护模式找不到目标 Skill 文件 | 提示用户确认文件名，给出模糊匹配候选 | 列出 `~/.workbuddy/skills/` 下的近似匹配 |
| Git 仓库未初始化（维护模式） | 自动 `git init` 并做初始提交 | 不阻塞进化循环 |

## 附录：9 维评分标准

> 用于进化循环的基线评估和双评委审查。每维 0-10 分，总分 90。

| # | 维度 | 0-3 分 | 4-6 分 | 7-8 分 | 9-10 分 |
|---|------|--------|--------|--------|---------|
| 1 | **可读性** | 结构混乱，术语不统一 | 有基本结构但部分模糊 | 结构清晰，术语统一 | 新人 10 分钟可完全理解 |
| 2 | **完整性** | 缺 YAML 或执行区 | 三区齐全但字段不足 | 三区齐全 + 自检清单 | 完全符合蓝图骨架全部要素 |
| 3 | **模块匹配度** | 多个 step 无工具调用 | 大部分有但 1-2 存疑 | 全部 step 工具匹配合理 | 工具选择精准，无冗余 |
| 4 | **错误处理** | 无任何异常处理描述 | 提到但无具体策略 | 关键 step 有异常分支 | 每个高风险 step 有降级策略 |
| 5 | **安全性** | 无安全约束 | 有基础约束未全覆盖 | 硬约束覆盖核心风险 | 三级风险定级完整 |
| 6 | **IO 闭合** | step 间输入输出断裂 | 基本闭合但 1-2 处隐式依赖 | 全部显式声明，链路可追踪 | 数据格式在各 step 间一致 |
| 7 | **反例覆盖** | 无任何反例 | 1-2 个但未覆盖高风险 step | 每个高风险 step 有反例 | 与全局反模式交叉验证无遗漏 |
| 8 | **约束一致性** | 约束间互相矛盾 | 无矛盾但缺优先级链 | 分类清晰 + 优先级链明确 | 含冲突消解规则 |
| 9 | **执行效率** | 冗余 step 2+ 个 | 有 1 个冗余 | step 数量合理，确认点适中 | 每步有实质产出，无流程 step |

## 已知局限

1. 生成的 Skill 是"骨架级"——具体工具调用细节需在实际使用中补全
2. 对含多分支决策树（10+ 路径）的复杂 Skill，建议先简化核心逻辑
3. 预检层搜索依赖关键词匹配，语义级近似可能漏掉
4. 进化循环的 9 维评分依赖 LLM 判断，非客观量化——双评委设计部分缓解但无法根除
5. 进化循环依赖 Git 做版本控制；如果 Skill 目录不在 Git 仓库中，回滚功能不可用
6. 单次进化只改 1 个维度——多维度优化需要多轮进化循环

## 维护指引

- **骨架扩展**：新增任务类型时，在「附录：5 种标准骨架模板」中追加，保持格式一致
- **反模式更新**：发现新的常见错误模式时，加入「不做什么」列表，保持 13 条的结构
- **评分标准迭代**：9 维评分标准如需调整，同步更新所有引用该标准的 Skill
- **跨平台迁移**：其他平台使用者保留预检层 + 骨架结构 + 进化循环，替换具体工具名即可
