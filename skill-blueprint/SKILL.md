# Skill Blueprint — 自然语言 → 工程 Skill 的自动翻译器

## 设计哲学

> **先搜再建，不重复造轮子。**
> 创建一个 Skill 前，先判断是否常见需求，再搜索是否已有可复用的方案。能用现成的就不写新的，能小改就不重写。

本 Skill 的核心任务：
1. 先搜：判断需求典型度 → 搜索本地/社区/开源 → 决定「直接用/改编/新建」
2. 再建：把模糊的自然语言意图翻译成清晰的执行步骤
3. 统一：用标准骨架规范确保不同人/agent 产出的 Skill 互相可读
4. 高效：一次推导 + 一次确认，不搞马拉松式交互

---

## 预检层：先搜再建（进入核心工作流前必执行）

> **双轨评估的内核**：优化现有流程（适配已有 Skill）vs 换替代方案（从零新建）。能复用时不新建——这是经济性原则。

### Layer 0: 典型度判断（1 秒决策）

判断这个需求是否「典型」——典型意味着很可能已有现成方案，值得花一轮去搜。

| 典型（值得搜） | 不典型（跳过搜，直接新建） |
|------------|---------------------|
| 信息检索、监控、对比、日报周报 | 高度个人化的业务流程 |
| 数据分析、格式转换、代码生成 | 内网/私有系统专用场景 |
| Agent/Skill/MCP 等通用热门话题 | 用户明确说了"按我这个独特的逻辑" |
| 跨平台可复用的通用任务 | 涉及个人密码/密钥的非标准化操作 |

**决策规则**：
- 典型 → 执行 Layer 1 搜索
- 不典型 → 直接跳到核心工作流 Step 0
- 拿不准 → 倾向于搜（多花一轮搜索比错过一个 80% 可复用的方案划算）

### Layer 1: 搜索匹配（3 级级联）

按优先级从近到远搜索，找到第一个可用结果就停：

| 级别 | 来源 | 搜索方式 | 耗时 |
|:--:|------|---------|:--:|
| 1 | **本地已安装 Skill** | 列出 `~/.workbuddy/skills/` 下的 Skill，匹配名称和描述 | 0 秒 |
| 2 | **GitHub 开源 Skill** | WebSearch：`github.com <需求关键词> skill SKILL.md agent_created` | 1 轮 |
| 3 | **通用 Web 搜索** | WebSearch：`<需求场景> skill workflow template` | 1 轮 |

**搜索技巧**：
- 用需求的核心动词+名词组合搜。例如"每日 GitHub 热榜监控"搜 `github trending monitor skill`
- 优先搜 `topic:skill` 标签的 repo
- 如果找到的 SKILL.md 结构完整、有 step 拆解 → 标记为候选

### Layer 1 结果处理

| 匹配度 | 判断标准 | 处理方式 |
|:------:|---------|---------|
| **完美匹配** | 功能覆盖 ≥ 90%，只需无或极微调 | 告知用户 → 拉取/安装 → **跳过核心工作流，结束** |
| **可改编** | 功能覆盖 50-90%，改成目标版本 < 新建的 30% 成本 | 展示给用户："找到一个 X 项目，跟你的需求 YY% 吻合，改 Z 个参数就能用。直接用还是新建？" |
| **无匹配** | 没找到或改造成本 > 新建 30% | 进入核心工作流 Step 0 |

**改编成本估算口诀**：
- 只改触发词/名称/配置 → < 10%，直接用
- 需要改 1-2 个 step 逻辑 → 10-30%，可改编
- 需要重写 3 个以上 step → > 30%，不如新建

## 核心工作流

### Step 0: 意图解译（0-1 轮）

从用户描述中提取核心要素。信息够就直接跳到 Step 1；信息缺则提 **最多 1 个确认问题**。

**提取维度**：

| 维度 | 提取内容 | 示例 |
|------|---------|------|
| 任务本质 | 一句话：这个 Skill 解决什么问题 | "帮用户对比两个手机哪个该买" |
| 触发条件 | 什么时候该加载 | "用户说'哪个好'、'选哪个'" |
| 工作步数 | 从意图推导需要几步 | "搜信息 → 对比 → 推荐" |
| 约束条件 | 格式/来源/交互/偏好约束 | "必须引用视频评测"、"不要超过1000字输出" |

**规则**：
- 以上信息用户至少提供了 3 项 → 直接进入 Step 1，不要追问
- 只缺 1 项且很关键 → 问 1 个问题。例如："你希望它在什么时候自动触发？（比如用户说了什么话就该加载）"
- 缺多项 → 不要列清单审问。用 1 个概括性问题收窄方向。例如："你希望这个 Skill 主要用来做什么类型的任务？信息搜索、分析决策、还是自动操作？"
- **反例**："你是做什么用的？触发条件是什么？有没有约束？输出什么格式？" ← 一次抛 3+ 个问题，绝对禁止

---

### Step 1: 骨架自动推导（不中断，一次性完整输出）

基于 Step 0 提取的意图，自动推导完整的骨架结构并一次性呈现给用户。

**推导规则**：

1. **Step 数量**：3-5 个。多了太重，少了不够。如果用户自己说了步骤数量，优先听用户的
2. **逻辑递进**：步骤之间是递进关系，不是并列堆砌——上一个 step 的输出是下一个 step 的输入
3. **拿不准时归类**：遇到分叉逻辑（"如果是A就做X，如果是B就做Y"），归到同一个 step 内处理，不要另开新 step
4. **用户定义的步骤优先**：用户如果在描述中已经说了"应该分三步"，直接采用他的划分

**骨架标准格式**：

```markdown
---
name: <skill-name>
description: |
  一句话说明做什么 + 触发场景 + 触发词。
  使用第三人称（"当用户...时，应使用本 Skill..."）。
agent_created: true
---

# <skill-name>

## 一句话概述
[从用户意图推导，1-2 句说清做什么]

## 触发条件
- 触发词：[词1, 词2, ...]  ← 要具体，结合场景；避免用"推荐""比较"这种太宽泛的词
- 触发场景：[场景描述]

## 执行流程

### Step 1: <阶段名>
- **目标**：[这一步要达成什么]
- **输入**：[需要什么信息/状态]
- **执行**：[具体怎么做，清晰到另一个 agent 不用猜]
- **输出**：[这一步产出什么]
- **验收**：[怎么判断做完了、做对了]
- **⚠️ 别犯**：[这一步最常见的错误/反例]

### Step 2: <阶段名>
...

## 全局约束
- [跨 step 的通用规则，如格式要求、语言风格、数据来源规范]

## 局限说明
- [当前已知的能力边界]
```

**关键质量标准**（生成时逐条自查）：
- ✅ 另一个 agent 读一遍就能执行——不需要猜"这一步到底要我做什么"
- ✅ "验收"不含糊——能量化的量化，不能量化的说清"什么样算完"
- ✅ 没有空洞词——不说"充分了解""全面分析""深入理解"，说"收集至少 X 个信源""列出 N 个以上差异点"
- ✅ 触发词结合场景——不是孤立的"推荐""比较"，而是"买XX哪个好""XX和XX怎么选"

---

### Step 2: Checkpoint 确认

把骨架完整展示给用户，附一句简明提示。**口头禅式模板**：

> "这是按你的需求推导的骨架，步骤和方向对吗？要改什么直接说。"

| 用户回应 | 处理方式 |
|----------|---------|
| "好"、"行"、"没问题"、"可以" | → 进入 Step 3，直接写入 |
| "XX改成YY"、"加一个步骤"、"去掉XX" | → 改完后再展示一次修改版，再确认 |
| "不确定这样行不行" | → 提供 2-3 个具体修改建议让他选 |
| "太复杂了" | → 合并步骤；"太简单了" → 拆分细化 |

**禁止**：确认时说"还有其他需要调整的吗？"然后列出所有可改项让用户选——这在用户只需要微调一步时会显得啰嗦。等用户自己说。

---

### Step 3: 写入文件

确认无误后，将最终内容写入 `~/.workbuddy/skills/<skill-name>/SKILL.md`。

写入时注意：
- YAML frontmatter 必须有 `agent_created: true`
- `description` 必须用第三人称（"当用户...时，应使用本 Skill..."），不是"你"开头
- 写入完成后告知用户文件路径，并用一句话总结这个 Skill 的核心能力

## 不做什么（反模式）

1. ❌ **不搜就建**——这是最大反模式。用户说了需求之后，第一反应不能是"好，我来推导骨架"。必须先过预检层
2. ❌ **搜到不好用还硬推**——搜索结果匹配度 < 50% 就说没找到，不要硬塞一个不相关的给用户
3. ❌ **把用户当需求分析师**——不要问"你的目标用户是谁"、"核心指标是什么"这类产品需求问卷。用户需要的是一个 Skill，不是一个 PRD
4. ❌ **多轮拉锯**——预检层最多 1 轮搜索 + 核心工作流最多 1 轮确认。不要把简单的事聊复杂了
5. ❌ **生成空壳**——每个 step 的"执行"部分必须有实质性内容，不能是"根据情况选择合适的工具"这种空话
6. ❌ **越俎代庖**——用户说"我就想按这 4 步来"，就直接按 4 步做，不要建议改成 3 步
7. ❌ **过度设计**——这个 Skill 是搭骨架的，不是写实现的。不要替用户决定每个 step 该调哪个 API、用什么函数

## 附录：典型骨架模板（供内部参考，不作为输出标准）

以下骨架模板用于在 Step 1 推导时快速匹配用户意图类型：

### 信息检索型
```
Step 1: 拆解需求 → Step 2: 多源搜索 → Step 3: 结构化整理 → Step 4: 输出交付
```

### 分析决策型
```
Step 1: 理清问题 → Step 2: 收集数据 → Step 3: 对比分析 → Step 4: 给出结论
```

### 内容生成型
```
Step 1: 明确规格 → Step 2: 素材准备 → Step 3: 生成输出 → Step 4: 质检验收
```

### 自动化操作型
```
Step 1: 确认目标 → Step 2: 预检环境 → Step 3: 执行操作 → Step 4: 验证结果
```

### 评估评审型
```
Step 1: 确定标准 → Step 2: 逐项评估 → Step 3: 汇总结论 → Step 4: 改进建议
```

## 已知局限

1. 生成的 Skill 是"骨架级"——step 内部的具体工具调用、代码逻辑、数据格式等细节，需要在实际使用中由用户和其他 Skill 协作补全
2. 对含多分支决策树（10+ 路径）的复杂 Skill，骨架生成可能不够精准，建议用户先用一句话简化核心逻辑
3. 不支持跨 Skill 编排——如果需要"一个 Skill 调用另一个 Skill"的编排逻辑，不在本 Skill 范围内
4. 预检层的搜索依赖关键词匹配，语义级近似（项目名和描述里不用标准术语但功能相同）可能漏掉
5. 典型度判断基于经验规则，边缘情况可能误判——拿不准时走"倾向于搜"策略，宁可多搜一轮不错杀一个候选
