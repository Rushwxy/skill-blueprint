# GitHub Trend Monitor — 智能趋势雷达

## 一句话概述
每天扫一遍 GitHub 热榜和关注人物动态，筛出跟 Skill 构建、Agent 自我进化最相关的高价值内容，同时保留全局热榜帮你打破信息茧房，触类旁通。

## 触发条件
- **触发词**："GitHub今天有什么"、"GitHub趋势"、"GitHub热榜"、"最近有什么新项目"、"看看关注的GitHub"、"GitHub日报"
- **触发场景**：每日例行检查 / 主动想了解行业动向 / 寻找新的 Skill 或 Agent 相关项目

---

## 执行流程

### Step 1: 确定关注清单
- **目标**：明确本次查询的范围——什么话题、哪些人
- **输入**：用户的配置偏好 + 当前查询意图
- **执行**：
  1. 优先读取 `~/.workbuddy/skills/github-trend-monitor/config.md` 中的关注清单
  2. 如果配置不存在，使用默认清单：
     - **核心话题**：AI agent, LLM skill, agent framework, self-improving AI, agentic workflow, MCP, tool-use, agent orchestration, multi-agent, AI coding
     - **关注人物**：karpathy（默认关注），其余由用户补充
     - **关注组织**：langchain-ai, microsoft/autogen, anthropics, google-deepmind（默认关注）
  3. 关注清单格式（config.md）：
     ```yaml
     topics: [skill, agent, self-evolution, agentic-workflow, mcp, tool-use, multi-agent, ai-coding]
     users: [karpathy]
     orgs: [langchain-ai, microsoft, anthropics, google-deepmind]
     ```
- **输出**：确定的关注清单（话题 + 人物 + 组织）
- **验收**：清单非空，且每个条目都有明确的关注理由
- **⚠️ 别犯**：第一轮就列一堆问题让用户填关注清单——先用默认清单跑起来，用户觉得不对自然会补充

### Step 2: 四轨数据采集
- **目标**：从 4 个角度覆盖 GitHub 数据——热榜、关注动态、主题搜索、全局趋势
- **输入**：Step 1 的关注清单
- **执行**：
  1. **GitHub Trending 热榜**：通过 WebSearch 获取 `github.com/trending` daily/weekly 榜单，筛选与核心话题相关的项目（下限 15 条）
  2. **关注人物/组织动态**：使用 GitHub MCP 工具拉取关注用户和组织近期公开活动——新 repo、star 的项目、release、commit
  3. **主题关键词搜索**：用核心话题关键词搜索 GitHub repos（按 stars/updated 排序，最近 7 天），每话题取前 5 条
  4. **全局热榜（无过滤）**：获取 GitHub Trending daily 全量榜单前 20 名，不做任何主题过滤，完整保留
- **输出**：4 轨原始数据池（去重、带时间戳、标注来源轨）
- **验收**：4 轨均有数据，总计原始条目 ≥ 40
- **⚠️ 别犯**：只拉 trending 不搜主题——高价值小众项目不在热榜上；搜得太多（100+ 条）——后续筛选压力太大；忘了全局热榜——它是打破信息茧房的关键

### Step 3: 智能筛选与排序
- **目标**：从原始数据池筛掉噪音，留下值得看的内容
- **输入**：Step 2 的 4 轨原始数据
- **执行**：
  1. **相关性打分**（0-5 分）：
     - 直接命中核心话题（skill/agent/self-evolution）→ +3
     - 间接相关（tool-use/MCP/workflow/agent-framework）→ +2
     - 来自关注人物/组织 → +1
     - 高热度（stars 近期暴涨或 trending 排名靠前）→ +1
  2. **跨域亮点识别**（仅对全局热榜中不相关的条目）：
     - 代表新兴范式或基础设施变化（新语言/框架范式、架构迁移信号、工具链变革）→ 标记 ⚡
     - 高热度但领域完全不同的 → 保留 5-8 条到"全局趋势信号"
  3. **噪音过滤**：去掉教程合集类（awesome-list 等）、纯文档翻译、已归档项目、低质量 demo、README 不完整的
  4. **精选排序**：TOP 10 按总分排序，全部要求分数 ≥ 3
- **输出**：精选项目列表（含分数、关联理由、链接）+ 全局趋势信号列表
- **验收**：TOP 10 中至少 5 个是 skill/agent 直接相关，每个项目有"为什么值得看"的理由；全局趋势信号 5-8 条，每条有一句话信号解读
- **⚠️ 别犯**：只看 star 数忽略主题匹配——star 多但不相关属于全局热榜的事；把 14 天前更新的项目排到前面——优先近期活跃

### Step 4: 结构化日报输出
- **目标**：生成一份快速可读的日报，重点突出、链接直达
- **输入**：Step 3 的精选列表 + 全局趋势信号
- **执行**：
  1. 按以下结构组织输出：
     ```markdown
     ## 🔥 GitHub 趋势日报 — YYYY-MM-DD
     
     ### 精选关注（TOP N）
     | # | 项目 | Stars | 为什么值得看 | 链接 |
     |---|------|-------|-------------|------|
     | 1 | ... | ... | ... | ... |
     
     ### 关注人物/组织动态
     | 来源 | 动作 | 相关项目 | 时间 |
     |------|------|---------|------|
     | ... | ... | ... | ... |
     
     ### 主题热榜速览
     > 按话题分组展示当前热度与代表项目
     
     ### 🌍 全局趋势信号（打破茧房）
     | # | 项目 | Stars | 领域 | 信号解读 |
     |---|------|-------|------|----------|
     | 1 | ... | ... | ... | ... |
     ```
  2. 每条精选项目一句话推荐理由（不是描述功能，是"为什么你应该关注"）
  3. 全局趋势信号解读要一针见血——"暗示 XXX 趋势"、"XX 领域范式迁移"、"基础设施层在变化"
  4. 如果某个板块当日无高价值内容，明确写"今日无高价值更新"而不是省略
- **输出**：格式化的日报摘要
- **验收**：读完后 2 分钟内能决定哪些项目要深入查看；每个链接可直达；全局趋势信号板块独立醒目
- **⚠️ 别犯**：输出成又臭又长的报告——日报是雷达，不是百科全书；漏掉链接——只说项目名不给 URL 等于没给；全局趋势板块写得太敷衍——它和精选板块同等重要

## 全局约束
- 数据来源优先用 GitHub MCP 工具（用户已连接），备选 WebSearch/WebFetch
- 每日数据必须标注日期，避免历史数据混淆
- 关注清单（config.md）如果用户更新，下次采集自动生效
- 全局热榜不做任何过滤——宁可多留一条噪音也不错杀一个信号

## 局限说明
- GitHub Trending 数据依赖 WebSearch 获取，非官方 API，可能有半天到一天的滞后
- 关注人物的非公开活动（private repo 的 star 等）无法追踪
- 主题匹配依赖关键词，语义级创新（项目描述里不用 agent 这个词但实际是 agent 框架）可能漏掉
- 全局趋势信号的解读带有推断性，不保证准确，但努力给出合理的信号分析
